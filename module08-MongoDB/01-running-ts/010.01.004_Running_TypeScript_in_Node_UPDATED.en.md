TypeScript helps you catch bugs early and write more predictable code. However, TypeScript's type annotations are not understood by JavaScript runtimes like browsers, and until recently not by Node.js either. That is why we usually need a build step to strip types and produce plain JavaScript. Fortunately, modern Node.js can now run `.ts` files directly. Starting in Node.js v22 this became available, and by Node.js v24 it is mature enough to use as a simple development workflow.

On the frontend, you have worked in environments where everything was preconfigured: path aliases, extensionless imports, hot reload, and seamless TypeScript support. In the backend, things are more hands-on. Here, you assemble your own toolchain.

The goal is to build a small, reusable Node.js + TypeScript scaffold without bundlers. In development, we will run `.ts` files directly with Node. For production, we will still compile to JavaScript with `tsc`.

## TypeScript 6.0 Compatibility Note

TypeScript 6.0 changed a few defaults that affect older Node + TypeScript setups:

- `baseUrl` is deprecated. If you use `paths`, put the full `./src/...` prefix directly inside each path mapping.
- `rootDir` is no longer inferred the old way. Set `"rootDir": "./src"` explicitly if you want `src/app.ts` to compile to `dist/app.js` instead of `dist/src/app.js`.
- `types` now effectively defaults to `[]`. For a Node project, add `"types": ["node"]`.

This guide already uses the TypeScript 6 compatible version of the config.

## Initialise Your Project

Create a new folder and initialise a Node.js project:

```sh
mkdir my-ts-app && cd my-ts-app
npm init -y
```

Since we are used to working with ES Modules, make sure you set `"type": "module"` in `package.json`.

## Install Dependencies

```sh
npm install -D typescript @types/node
```

- `typescript`: the TypeScript compiler
- `@types/node`: type definitions so TypeScript understands Node.js APIs
- `-D`: installs them as development dependencies

## Create Additional Files and Directories

Create a `src` directory:

```sh
mkdir src
```

Then create the project files:

```sh
touch .gitignore tsconfig.json src/app.ts
```

## Example `package.json`

```json
{
  "name": "my-ts-app",
  "version": "1.0.0",
  "main": "app.ts",
  "type": "module",
  "scripts": {
    "dev": "node --watch src/app.ts",
    "prebuild": "rm -rf dist",
    "build": "tsc",
    "prestart": "npm run build",
    "start": "node dist/app.js"
  },
  "devDependencies": {
    "@types/node": "^25.5.0",
    "typescript": "^6.0.2"
  }
}
```

## Understanding Your Dev and Build Scripts

- `dev`: runs your `.ts` file directly with Node.js during development
- `build`: compiles TypeScript into JavaScript inside `dist/`
- `start`: runs the compiled JavaScript from `dist/app.js`
- `prebuild`: removes the previous `dist/` folder before each build
- `prestart`: automatically runs `build` before `start`

## `tsconfig.json`

The `tsconfig.json` file tells TypeScript how to interpret and compile your code. Here is a sensible default for a modern Node.js project that is compatible with TypeScript 6.0:

```json
{
  "compilerOptions": {
    /* Base Options */
    "esModuleInterop": true,
    "lib": ["ESNext"],
    "target": "ESNext",
    "types": ["node"],
    "skipLibCheck": true,
    "allowJs": true,
    "resolveJsonModule": true,
    "moduleDetection": "force",
    "isolatedModules": true,
    "verbatimModuleSyntax": true,

    /* Strictness */
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,

    /* Node Options */
    "allowImportingTsExtensions": true,
    "rewriteRelativeImportExtensions": true,
    "module": "preserve",
    "noEmit": false,
    "rootDir": "./src",
    "outDir": "dist",

    /* Paths */
    "paths": {
      "#*": ["./src/*"]
    }
  },
  "include": ["src"]
}
```

### Why These Settings Matter

- `"types": ["node"]`: required in TypeScript 6 so Node globals and modules are available to the compiler
- `"rootDir": "./src"`: keeps your build output flat, so `src/app.ts` becomes `dist/app.js`
- `"paths": { "#*": ["./src/*"] }`: replaces the old `baseUrl` approach
- `"module": "preserve"` and `"verbatimModuleSyntax": true`: keep modern ESM import/export syntax as-is
- `"rewriteRelativeImportExtensions": true`: helps emitted JavaScript use correct file extensions

## About `#` Imports

Why use `#` instead of `@`? Because that matches Node.js subpath imports.

Setting up aliases in Node.js is a two-step process:

1. Tell TypeScript how to resolve them in `tsconfig.json`
2. Tell Node.js how to resolve them at runtime in `package.json`

Example:

```json
"imports": {
  "#utils": {
    "development": "./src/utils/index.ts",
    "default": "./dist/utils/index.js"
  }
}
```

If you use aliases in development, update your `dev` script too:

```json
"scripts": {
  "dev": "node --watch --conditions development src/app.ts",
  "prebuild": "rm -rf dist",
  "build": "tsc",
  "prestart": "npm run build",
  "start": "node dist/app.js"
}
```

Now Node will resolve `#utils` to the TypeScript file in development and to the compiled JavaScript file in production.

## Create Your First Script

Create `src/app.ts`:

```ts
const message = "Hello from TypeScript!";
console.log(message);
```

Run it in development:

```sh
npm run dev
```

Node strips the TypeScript types at runtime. It does not type-check your code. Your editor and `tsc` are still responsible for that.

To build for production:

```sh
npm run build
```

Then start the compiled app:

```sh
npm start
```

## Actually Using an Alias

Create `src/utils/index.ts`:

```ts
export function pickRandom<T>(arr: T[]): T | undefined {
  if (arr.length === 0) return undefined;
  const index = Math.floor(Math.random() * arr.length);
  return arr[index];
}
```

Then in `src/app.ts`:

```ts
import { pickRandom } from "#utils";

console.log(pickRandom([1, 2, 3, 4, 5]));
console.log(pickRandom(["apple", "banana", "cherry"]));
console.log(pickRandom([]));
```

This should work in both development and production, as long as:

- `tsconfig.json` contains the `paths` mapping
- `package.json` contains the matching `imports` mapping
- your `dev` script uses `--conditions development`

## Summary

This setup gives you:

- native `.ts` execution in Node.js during development
- proper type-checking through TypeScript and your editor
- a production build that emits plain JavaScript
- TypeScript 6.0 compatible path aliasing without `baseUrl`

You now have a modern scaffold for backend development with Node.js and TypeScript.
